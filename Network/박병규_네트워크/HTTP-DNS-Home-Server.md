# **Ⅰ. WEB2_HTTP**<br/> 
## **1. HTTP 소개**<br/>
- ### HyperText Transfer Protocol (HTTP)
- ### HTTP
    - #### Request message (요청)
    - #### Response message (응답)

## **2. Request message**<br/>
- ### Request Line
    - #### GET 
        ##### 메소드 / 웹브라우저와 웹서버가 어떤 방식으로 통신할것 인가?

- ### Request Headers
    - #### Host
        ##### 인터넷에 연결되어 있는 컴퓨터를 식별하는 것
    
    - #### User-Agent
        ##### User-Agent = 웹브라우저
        ##### 자신이 사용하고 있는 브라우저, 운영체제를 보여준다.
        
    - #### Accept-Encoding
        ##### 데이터양이 많을때 압축하여 전송하면 웹브라우저가 자동으로 압축을 풀어서 처리
        ##### 웹브라우저가 어떤 압축방식을 지원하는 지 표시
    
    - #### If-Modified-Since
        ##### 마지막으로 다운로드 받은 날짜를 표시해 두고 웹서버는 이를 비교하고 더 최신인 파일을 전송해 준다.


## **3. Response message**<br/>
- ### status (version - status code - phrase)
    - #### status code
       - ##### 1xx - Informational responses 
       - ##### 2xx - Success<br/>
            ##### (200 OK)
       - ##### 3xx - Redirection
       - ##### 4xx - Client errors
            ##### (404 Not Found)
       - ##### 5xx - Server errors
- ### Headers
    - #### Last-Modified
        - ##### 마지막 수정날짜  
    - #### Content-Encoding
        - ##### 사용한 압축방식 표시
    - #### Content-Length
        - ##### 응답하는 컨텐츠의 크기(byte)
    - #### Content-Type
        - ##### 웹 서버의 응답하는 컴퓨터 언어를 표시한다.

## **4. HTTP / HTTPS**<br/>
- ### HTTP 안전하지 않다.
- ### HTTPS 안전하다.

# **Ⅱ. WEB2_Domain Name System** <br/>

## **1. Domain Name System 소개**<br/>
- ### Domain Name System (DNS)
    - #### 호스트의 도메인 이름을 호스트의 네트워크 주소로 바꾸거나 그 반대의 변환을 수행하 수 있도록 하기 위해 개발한 것이다.
    - #### 특정 컴퓨터(또는 네트워크로 연결된 임의의 장치)의 주소를 찾기 위해 사람이 이해하기 쉬운 도메인 이름을 숫자로 된 식별 번호(IP 주소)로 변환해 준다. 

## **2. DNS의 태동**<br/>
- ### Stanford Research Institute
    - #### 과거에는 이 곳에서 전세계 모든 hosts 파일을 관라하였다.
    - #### 이곳의 hosts 파일을 다운받아 접속할수 있었다.

- ### Stanford Research Institute의 문제점
    - #### Stanford Research Institute 에서 hosts 파일을 갱신하지 않으면 새로운 파일을 사요하지 못하였다. 
    - #### hosts 파일 갱신을 수작업으로 하여 수많은 시간과 비용이 들어간다.
    - #### 하나의 파일에 모든 hosts 파일이 들어가면 언젠가는 한계가 생긴다.

- ### 이러한 문제로 인해 새로운 시스템이 생기는데 이것이 Domain Name System(DNS) 이다.


## **3. DNS의 원리**<br/>
- ### Domain Name System Server (DNS Server)
- ### Domain Name System Server
    - #### 도메인의 이름과 도메인의 IP주소를 저장하고 기억해두는 역할을 한다.

- ### Domain Name System Server의 원리 
    - #### 1. 개인컴퓨터가 DNS Server에 도메인 이름을 물어본다.
    - #### 2. DNS Server는 개인컴퓨터에게 도메인이름으로 저장되있는 IP주소를 전달해 준다.
    - #### 3. 개인 컴퓨터는 받은 IP주소로 접속할수 있다.

- ### Stanford Research Institute보다 좋은점
    - #### 과거는 전화를 걸어 평일에만 IP주소를 받을수 있었지만 현재는 자동화로 빠르고 자유롭게 받을수 있게 되었다.
    - #### 행정절차가 간소화되고, 신속해졌다.
    - #### DNS Server는 IP주소가 변경되는 순간 즉시 모든 IP주소 사용 컴퓨터는 자동 변경될수 있다.


## **4. Public DNS**<br/>
- ### 통신사에서 제공하는 DNS
- ### Google의 public DNS
    - #### 8.8.8.8
    - #### 8.8.4.4


## **5. 도메인 이름의 구조**<br/>

    | 주소 | 기능 | 
    |:----------|      :----------:|
    | . | Root | 
    | com | Top-level |

- ### Root -> Top-level -> Second-level -> sub

## 6. 도메인 이름 등록 과정과 원리 <br/>
- ### Registrant -> Registrar -> Registry -> ICANN

    | 도메인 등록 명 | 별명 | 
    |:----------|      :----------:|
    | Registrant | 등록자 | 
    | Registrar | 등록대행자 | 
    | Registry | 등록소 | 

    - #### 등록자 -> 등록대행자
        - 도메인주소와 이름을 알려준다.
    - #### 등록대행자 -> 등록소
        - 등록소는 Top-level domain에 도메인주소와 이름을 저장해둔다. 

## 7. nslookup<br/>
- ### Name Server Lookup (nslookup)
- ### 도메인 네임을 얻거나 IP주소 매핑 또는 다른 특정한 DNS 레코드를 도메인 네임 시스템(DNS)에 질의 할때 사용한다.

## 8. DNS record & CNAME<br/>
- ### DNS record
    - #### DNS서버에 저장하는 도메인의 이름의 정보 한건
- ### DNS 코드
    | 코드명 | 기능 | 
    |:----------|:----------:|
    | A | IP주소(Address) | 
    | NS | 관리 Name Server(Name Server) | 
    | CNAME | 도메인의 별명 (Canonical Name) |

# **Ⅲ. WEB2_Home server** <br/>
## **1. Router**
- ### Router = 공유기
    - #### WAN 과 LAN 사이에서의 중계자 역할을 하는 것이다. 

- ### WAN
    - #### 통신사와 인터넷 연결이 가능하게 해준다.
- ### LAN
    - #### 여러 장비들에게 인터넷 연결을 나누어 준다.
- ### 안테나
    - #### 무선 기기(휴대폰)을 연결해 준다.

- ### Public IP Address & Private IP Address
    - #### Public IP Address
        - ##### 광역네트워크 안에서 WAN에게 주는 IP
    - #### Private IP Address
        - ##### 공유기 안에서 여러 장비에게 나누어 주는 IP
- ### 사설 IP 범위 / 개수 
    | IP범위 | 개수 | 
    |:----------|:----------:|
    | 10.0.0.0 ~ 10.255.255.255  | 16,777,216개 | 
    | 172.16.0.0 ~ 172.31.255.255 | 1,048,576개 | 
    | 192.168.0.0 ~ 192.168.255.255 | 65,536개 |

## **2. NAT**
- ### Network Address Translation (NAT)

- ### NAT의 역할
    - #### 1. private IP address 의 요청을 기록한다.
    - #### 2. private IP address 를 public IP address로 변환한다.
    - #### 3. 응답 받은 정보를 요청자가 맞는지 확인한뒤 private IP address에게 전달해준다.

## **3. Port**
- ### 데이터를 받을 프로세스가 어떤것인지를 알아야 하는데 이때 사용하는 식별자를 Port Number (포트 번호)라고 한다.

- ### 포트번호 구분
    | 범위 | 범주 | 
    |:----------|:----------:|
    | 0 ~ 1023  | Well-known ports | 
    | 1024 ~ 49151 | Registered pots | 
    | 49152 ~ 65535 | Private ports |

- ### 포트번호 표기법
    - #### 포트번호는 IP주소 뒤에 :(콜론)을 사용하여 표기한다.

## **4. Port Forwarding**
- ### Port Forwarding 또는 Port Mapping 이라고 한다.

- ### Port Forwarding은 컴퓨터 네트워크에서 네트워크 게이트웨이를 가로지르는 동안 하나의 IP주소와 포트 번호 결합의 통신 요청을 다른곳으로 넘겨주는 네트워크 주소 변환의 응용이다.

- ### Port Forwarding의 목적
    - #### 인터넷 상의 컴퓨터가 LAN 내에 위치한 특정 컴퓨터에 연결할수 있게 한다.

## **5. Dynamic IP Address & Static IP Address**
- ### Dynamic IP Address (유동 IP 주소)
    - #### IP주소의 부족을 해결하기위한 방안이다.
    - #### 오랫동안 IP주소를 사용하지 않을시 ISP에서 IP주소를 회수하고, 다른사람에게 IP주소를 할당한다. 다시 돌아올시 새로운 IP주소를 할당한다.

- ### Static IP Address (고정 IP 주소)
    - #### ISP에 의해 해당 인터넷 사용자 전용으로 부여되는 인터넷 주소이다.
    - #### 특별한 사유가 없는 한 변하지 않고 사용되는 주소를 말한다.

## **6. DHCP**
- ### Dynamic Host Configuration Protocol (DHCP)
    - #### 네트워크 안에 컴퓨터에 자동으로 네임 서버 주소, IP주소, 게이트웨이 주소를 할당해주는 것을 의미하고, 해당 클라이언트에게 일정 기간 임대를 하는 동적 주소 할당 프로토콜이다.

- ### DHCP Client & DHCP Server
    - #### DHCP Client - 인터넷 사용자 사용
    - #### DHCP Server - 라우터(공유기)가 사용
    - #### HCP Client는 DHCP Server에게 IP주소를 요청하고 이를 얻을수 있다.

- ### DHCP 장점
    - #### PC의 수가 많거나 PC자체 변동사항이 많은 경우 IP설정이 자동으로 되기 때문에 효율적으로 사용 가능하고, IP를 자동으로 할당해주기 때문에 IP 충돌을 막을수 있다.

- ### DHCP 단점
    - #### DHCP 서버에 의존되기 때문에 서버가 다운되면 IP할당이 제대로 이루어지지 않는다.

# **Ⅳ. 반드시 알고 넘어가야 할 웹 기술 기초편** <br/>

## **1. 웹의 탄생과 발전**
- ### 웹의 탄생
    - #### Web (웹) = World Wide Web (월드 와이드 웹) = www / web
      
    - #### 인터넷상의 정보를 하이퍼텍스트 방식고과 멀티미디어 환경에서 검색할 수 있게 해주는 정보검색 시스템이다.
    - #### 하이퍼 텍스트 형식으로 표현된 인터넷상의 다양한 정보를 효과적으로 검색하는 시스템으로 전 세계적으로 가장 널리 보급 되어있다.
    
- ### 웹의 역사
    - #### 팀 버너스리 등의 제안으로 시작되어 연구 개발되었다. 원래는 세계의 여러 대학과 연구기관에서 일하는 물리학자들 상호간의 신속한 정보교환을 위해  하이퍼텍스트 중심으로 개발되었다.
  
- ### World Wide Web(월드 와이드 웹)의 3대 요소
    - #### 통일된 웹 자원의 위치 지정 방법 - URL
    - #### 웹의 자원 이름에 접근하는 프로토콜 - HTTP
    - #### 자원들 사이를 쉽게 항해 할 수 있는 언어 - HTML

- ### 하이퍼 텍스트
    - #### 하이퍼 텍스트는 웹 브라우저라 불리는 프로그램을 통해 웹 서버에서 문서나 이미지등의 정보를 읽어들여 컴퓨터 모니터에 출력하는 형태이다.

## **2. URL이란?**
- ### Uniform Resource Locator(URL)
    - #### 네트워크 상에서 자원이 어디 있는지를 알려주기 위한 규약이다. 즉 컴퓨터 네트워크와 검색 매커니즘에서의 위치를 지정하는 것이다. URL은 웹 사이트 주소뿐만 아니라 컴퓨터 네트워크상의 자원을 모두 나타낼 수 있다.
 
- ### URL의 구조
    - #### URL에는 Schem(스킴), Host(호스트), Port(포트), Path(패스), Query(쿼리), Fragment(프래그먼트)
    - #### URL에 항상 포함되는 것은 chem(스킴), Host(호스트), Path(패스)
      
    - #### Schem - 사용할 프로토콜
    - #### Host - 호스트명이나 IP주소
    - #### Port - 포트번호
    - #### path - 경로

## **3. HTTP 프로토콜**
- ### Hypertext Transfer Protocol (HTTP)
    - #### 인터넷상에서 데이터를 주고 받기 위한 서버/클라이언트 모델을 따르는 프로토콜이다. 
    - #### HTTP는 어떤 종류의 데이터든지 전송할 수 있도록 설계돼 있다. 
    - #### TTP로 보낼 수 있는 데이터는 HTML문서, 이미지, 동영상, 오디오, 텍스트 문서 등 여러종류가 있다.

- ### 작동방식
    - #### 클라이언트에서 요청(request)를 보내면 서버는 요청을 처리해서 응답(response)한다.
    - #### 웹서버는 보통 표준포트 80번 포트로 서비스한다.

- ### HTTP의 진화
    - #### HTTP/0.9 - 원-라인 프로토콜
       - #### HTTP 초기 버전에는 버전 번호가 없었습니다. 요청은 단일 라인으로 구성되며 리소스에 대한 경로로 가능한 메서드는 GET이 유일했습니다.
     
    - #### HTTP/1.0 - 확장성 만들기
       - ##### 브라우저와 서버 모두 좀 더 융통성을 가지도록 빠르게 확장 되었다. HTML 파일들 외에 다른 문서들을 전송하는 기능이 추가
     
    - #### HTTP/1.1 - 표준 프로토콜
       - ##### 모호함을 명확하기 하고 많은 개선 사항들을 도입하였다. 언어, 인코딩 혹은 타입을 포함한 컨텐츠 협상이 도입되어, 클라이언트와 서버로 하여금 교환하려는 가장 적합한 컨텐츠에 대한 동의를 가능케 했다
     
    - #### HTTP2 - 더 나운 성능을 위한 프로토콜
       - ##### 텍스트가 아닌 바이너리 프레임으로 구성, 다수의 요청/응답을 동시에 처리 가능/ 헤더의 크기 80% 정도로 줄임/ 중요한 리소스의 처리 지연을 방지


## **4. 쿠키와 세션**
- ### HTTP의 특징과 쿠키와 세션을 사용하는 이유
    - #### HTTP 프로토콜의 특징이자 약점을 보완하기 위해서 사용한다.

- ### 쿠키 (Cookie)
    - #### 쿠키는 클라이언트(브라우저) 로컬에 저장되는 키와 값이 들어있는 작은 데이터 파일입니다.
    - #### 사용자 인증이 유효한 시간을 명시할 수 있으며, 유효 시간이 정해지면 브라우저가 종료되어도 인증이 유지된다는 특징이 있습니다.

- ### 쿠키의 구성 요소
    - #### 이름 : 각각의 쿠키를 구별하는 데 사용되는 이름
    - #### 값 : 쿠키의 이름과 관련된 값
    - #### 유효시간 : 쿠키의 유지시간
    - #### 도메인 : 쿠키를 전송할 도메인
    - #### 경로 : 쿠키를 전송할 요청 경로

- ### 세션
    - #### 세션은 쿠키를 기반하고 있지만, 사용자 정보 파일을 브라우저에 저장하는 쿠키와 달리 세션은 서버 측에서 관리합니다.
    - #### 서버에서는 클라이언트를 구분하기 위해 세션 ID를 부여하며 웹 브라우저가 서버에 접속해서 브라우저를 종료할 때까지 인증상태를 유지합니다.

- ### 세션의 특징
    - #### 각 클라이언트에게 고유 ID를 부여
